Section 02

The 4/42 Architecture

Lume Scan is powered by a deterministic diagnostic engine — a software architecture where four irreducible flow primitives govern forty-two operational nodes. This is not a marketing construct; it is a mathematical architecture with formal properties.

Four Flow Primitives

Every vehicle's engine state can be decomposed into four independent measurement domains:

PrimitiveNodesDomain
Combustion EfficiencyCE1–CE12 (12)Fuel trims, O2 sensors, MAF/MAP, timing advance, catalyst efficiency
Thermal RegulationTR1–TR10 (10)Coolant temp, intake air temp, catalyst temp, thermal drift
Mechanical IntegrityMI1–MI10 (10)RPM stability, misfire counts, engine load, vibration signatures
Electrical SystemsES1–ES10 (10)Battery voltage, alternator output, sensor reliability, control module health

Why 42 Nodes?

The number is not arbitrary. Forty-two is the minimum node count required to achieve cross-validated consensus across all four primitives while maintaining single-cycle deterministic evaluation. Each node holds a normalized state value in the range [-1.0, +1.0]. Three threshold bands classify every node as nominal, caution, or critical. A deterministic engine selects one of five operating modes based on the aggregate state.

Five Operating Modes

  1. SAFETY — Any critical-band node detected. Immediate alert. No interpretation.
  2. RECOVERY — Multiple caution-band nodes across two or more primitives.
  3. BALANCE — Mixed state. Some nodes optimal, others degrading.
  4. EFFICIENCY — All nodes nominal. Fuel coaching active for maximum economy.
  5. STABILITY — Default. Steady-state monitoring. Baseline establishment.
Determinism Guarantee

Given the same 42 input signals, Lume Scan will produce the identical diagnostic output on any device, at any time, regardless of hardware. This property is mathematically provable and independently verifiable via the Trust Layer Ledger.

Section 04

Fuel Coaching Engine

Lume Scan's fuel coaching system is not a simple MPG calculator. It is a passive real-time feedback loop that monitors combustion efficiency across 12 nodes simultaneously and provides actionable, second-by-second guidance through visual and optional audio cues.

How It Works

  1. Baseline Establishment — During the first 5 minutes of driving, the engine establishes a fuel consumption baseline by averaging MAF, fuel trims, engine load, and throttle position across steady-state intervals.
  2. Efficiency Scoring — Each second, the engine computes an efficiency score from 0–100 by comparing current combustion parameters against the established baseline. Scores above 75 indicate optimal driving.
  3. Visual Feedback — A color-coded indicator (green/yellow/red) provides instant feedback without requiring the driver to read numbers or interpret gauges.
  4. Audio Coaching — Optional audio tones alert the driver when acceleration patterns, idling duration, or throttle behavior fall outside the efficiency envelope.
  5. Session Summary — At trip end, Lume Scan produces a fuel efficiency report with estimated savings, efficiency score, and specific improvement recommendations.

Savings Methodology

Estimated savings are calculated from measurable efficiency gains in three categories:

CategoryMechanismAnnual Savings
Driving BehaviorThrottle smoothing, idle reduction, cruise optimization$100–$180
Maintenance TimingO2 sensor degradation, air filter, spark plugs detected before failure$50–$90
Fuel Trim MonitoringRich/lean drift alerts enabling proactive service$30–$50

Total estimated annual savings: $180–$320 in fuel costs alone. Combined with preventive maintenance savings (Section 5), total value reaches $800–$2,400+ per year.

Section 05

Predictive Maintenance

Lume Scan detects component degradation before it triggers a diagnostic trouble code (DTC) or illuminates the check engine light. This is achieved through continuous threshold monitoring across all 42 signals — not through machine learning or statistical prediction.

Detection Categories

  • Catalyst Degradation — Cross-validates upstream vs. downstream O2 sensors to detect efficiency loss weeks before P0420/P0430 triggers. Estimated savings: $800–$1,500 (early catalyst replacement vs. emergency failure).
  • Fuel System Drift — Monitors short-term and long-term fuel trims simultaneously. Persistent drift beyond Â±8% triggers an alert before the ECU sets a misfire or fuel system DTC. Savings: $200–$600.
  • Cooling System Degradation — Tracks coolant temperature stabilization time and thermal cycling patterns. Detects failing thermostats, low coolant, or head gasket seepage. Savings: $150–$3,000+.
  • Ignition System Wear — RPM stability analysis and misfire pattern detection identify coil pack and spark plug degradation. Savings: $100–$400.
  • Emissions Readiness — Monitors all 8 OBD-II readiness monitors continuously. Alerts if any monitor resets or fails, preventing surprise state inspection failures.
  • Electrical System Health — Battery voltage trend analysis detects alternator degradation and parasitic drain patterns before a no-start event.
No False Positives by Design

Unlike AI-based predictive systems, Lume Scan's threshold-based detection produces zero false positives. An alert is issued only when a measured value crosses a defined threshold — not when a probability model suggests it might. If the engine says "caution," the sensor data proves it.

Section 06

Security & Verification

Every Lume Scan diagnostic session produces a cryptographically verifiable record. This is not optional — it is architecturally embedded in every scan.

Encryption

  • At Rest — AES-256-GCM encryption for all stored diagnostic data on the device.
  • In Transit — TLS 1.3 for any network communication (license verification, optional telemetry export).
  • Packet Signing — Ed25519 digital signatures on every diagnostic packet, ensuring data integrity from adapter to output.

Trust Layer Ledger (TLL) Hallmark

Every completed scan produces a SHA-256 hash of the full diagnostic output. This hash is anchored to the Trust Layer Ledger — a public, immutable verification chain operated by DarkWave Studios.

Verified but Private

The TLL stores only the cryptographic hash — not the diagnostic data itself. Anyone can verify that a scan result has not been altered by comparing their copy's hash against the ledger. But no one can extract the original data from the hash. Your vehicle data stays on your device. The ledger proves it hasn't been tampered with.

The TLL Explorer

The TLL Explorer at lumescan.tech provides public, real-time access to all anchored scan events. Each entry shows:

  • Timestamp of the scan event
  • SHA-256 hash of the diagnostic output
  • Engine mode at time of scan (SAFETY / RECOVERY / BALANCE / EFFICIENCY / STABILITY)
  • Verification status (anchored / pending)

No VIN, no location, no personal data is exposed. The explorer demonstrates that the verification infrastructure is live, operational, and independently auditable.

Read-Only Architecture

Lume Scan operates in read-only mode at the firmware level. The application reads OBD-II signals via standard Mode 01 (live data), Mode 02 (freeze frame), Mode 03 (stored DTCs), Mode 07 (pending DTCs), and Mode 09 (VIN). Code clearing (Mode 04) requires explicit user confirmation and a separate acknowledgment flow. No write commands are sent to the ECU without direct user initiation.

Section 08

Hardware Compatibility

Lume Scan supports any ELM327-compatible OBD-II adapter via Bluetooth Low Energy (BLE). The application uses the Web Bluetooth API (PWA) and native BLE stack (APK) to communicate with the adapter.

Supported Adapters

  • Any ELM327 v1.5+ BLE adapter
  • Any ELM327 v2.1+ Classic Bluetooth adapter (APK only)
  • WiFi ELM327 adapters (APK only, TCP connection)

Vehicle Compatibility

Lume Scan works with any vehicle manufactured after 1996 (OBD-II mandate year) that supports the standard OBD-II diagnostic protocols:

  • CAN (ISO 15765) — Most vehicles 2008+
  • ISO 9141-2 — European vehicles
  • ISO 14230 (KWP2000) — Asian vehicles
  • SAE J1850 (PWM/VPW) — Older American vehicles
